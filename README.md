# HLTV 选手数据采集与结构化处理流程

## 工作流程概述

1. **批量自动打开选手页面**：根据 players.csv 自动用 Edge 打开所有选手的 HLTV 页面。
2. **手动保存页面 HTML**：在浏览器中用自制插件一键保存页面 body HTML 到同一文件夹（如 E:/下载）。
3. **批量结构化数据**：用数据爬虫（data_processor.py）批量处理保存下来的 HTML，生成结构化数据。

---

## 操作步骤

### 1️⃣ 批量自动打开所有选手页面

在 `data_processor_test` 目录下运行：

```bash
python batch_workflow.py
```

脚本会自动读取 `players.csv`，依次用 Edge 打开每个选手页面。

### 2️⃣ 手动保存页面 body HTML

1. 在浏览器中切换到每个打开的页面。
2. 点击自制插件按钮（HLTV Body Saver），将页面 body HTML 保存到同一文件夹（如 E:/下载）。
	- 插件会自动用 url 末尾命名文件。

### 3️⃣ 批量结构化数据

继续运行 `batch_workflow.py`，或单独运行：

```bash
python data_processor.py "E:/下载/xxx.html"
```

脚本会自动处理所有 HTML 文件，输出到 `processed_data/选手名/` 目录下。

---

## 相关文件说明

- `players.csv`：选手列表，包含 name 和 HLTV 页面 url。
- `batch_workflow.py`：自动化批量打开页面、批量处理 HTML 的主流程脚本。
- `data_processor.py`：结构化处理单个 HTML 文件，生成 csv/json。
- `hltv_body_saver/`：浏览器插件源码，点击按钮即可保存页面 body HTML。

---

## 注意事项

- 插件保存的 HTML 文件建议统一放在 E:/下载 目录，便于批量处理。
- 插件和脚本无需 Playwright profile，所有操作均在真实浏览器和人工插件中完成。
- 如需调整 HTML 保存目录或文件名规则，可修改插件或 batch_workflow.py。

#  **数据平权：大众CS选手实力评级 EquiRating**

## 一、项目基本信息

- **项目名称**：CS选手能力分析模型（非商用）
    
- **项目周期**：1-2个月（灵活调整）
    
- **核心目标**：构建能基于HLTV和比赛数据客观评价CS选手的算法模型
## 二、项目背景

目前HLTV年度TOP20选手的评选机制包含两个部分：

1. **客观数据部分**：Rating、KAST、ADR等可量化的比赛数据
    
2. **主观评价部分**：评委基于选手在精英级赛事（如Major淘汰赛阶段）中的关键时刻表现、对比赛走势的影响力等非量化因素进行的主观评估
    

普通玩家/非顶尖职业选手缺乏被专业评委观察和主观评价的机会，只能获得基础比赛数据。这导致现有纯数据驱动的评价体系无法完整还原TOP20评选的逻辑，**对普通玩家的评价存在偏见和局限性**。

## 三、项目目标

开发一个能够模拟“主观评价部分”的模型，使基于普通数据的评价体系更接近专业评委的综合评判逻辑。

## 四、项目产出

一个面向普通玩家的评价模型，具备以下核心功能：

**基础功能**：相对评判

- 输入多个玩家的普通比赛数据（如Rating、KAST、ADR等）
    
- 输出这些玩家之间的相对实力排名（类似TOP榜单的排序效果）
    
- 排名逻辑应比纯数据排名更接近融合了主观评价的TOP20评选结果

## 五、项目流程
1. 数据层
	1. 收集 近10年TOP榜单的数据
		1. TOP榜单选手
		2. 选手当年的数据
	2. 少量玩家数据
		1. 从D+到S的数据
2. 处理层
	1. 数据处理
	2. 整理共有数据属性
	3. 特征处理
3. 模型层
	1. 基于pytorch开发
		1. 特征加权的基线模型
		2. 基于对比学习的模型
		3. 带Attention模块的模型
	2. 模型验证于评估体系
4. 应用层
	1. 对于每一年的TOP预测准确度达到70%
	2. 对于玩家间的水平预测准确度达到70%

## 六、技术难点
- 获取数据：网络爬虫，现成数据集
- 模型训练：pytorch的搭建
- 评估体系：如何分离出主观的评价部分
